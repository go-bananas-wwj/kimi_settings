# Product Lens — Think Before You Build（Kimi 适配版）

> 产品诊断专用，不直接输出可落地的技术规格文档。
> 如果需要把诊断结果转成 PRD 或需求文档，请在诊断完成后明确提出。

## 触发方式

在 Kimi 中直接描述你的场景即可：

- "帮我诊断一下这个项目"
- "用 product-lens 审查一下 /workspace/xxx"
- "分析一下这个产品方向的优缺点"
- "我卡住了，不知道该做哪个功能"

## 何时使用

- 开始做任何功能之前 —— 先验证"为什么"
- 每周产品复盘 —— 我们在做对的事吗？
- 在多个功能间犹豫不定
- 上线前 —— 快速 sanity check 用户旅程
- 把一个模糊想法转成产品 brief，再进入工程排期

## 工作模式

### 模式 1：Product Diagnostic（产品诊断）

像 YC office hours 一样，自动提出尖锐问题：

```
1. 这是给谁做的？（具体的人，不是"开发者"这种泛泛标签）
2. 痛点是什么？（量化：多频繁、多严重、他们现在怎么凑合的？）
3. 为什么是现在？（发生了什么变化，让这个需求变得可行/必要？）
4. 10 星版长什么样？（如果钱和时间无限，最完美的形态是什么？）
5. MVP 是什么？（最小可验证单位，能证明这个 thesis 的最小东西）
6. 反目标是什么？（你明确不做什么？）
7. 怎么知道它成功了？（指标，不是感觉）
```

输出：一份 `PRODUCT-BRIEF.md`，包含答案、风险评估和 go/no-go 建议。

如果结论是"值得做"，下一步应该是撰写正式的 PRD/需求文档，而不是继续开会讨论。

### 模式 2：Founder Review（创始人视角审查）

用创始人眼光审视当前项目，找闪光点和浪费：

```
1. 阅读项目文档：README、AGENTS.md、package.json、最近提交记录
2. 推断：这个项目到底想成为什么？
3. 评估产品-市场契合信号（0-10 分）
   - 使用增长轨迹
   - 留存信号（重复贡献者、回访用户）
   - 收入信号（定价页、计费代码、支付集成）
   - 竞争护城河（什么东西别人抄不走？）
4. 识别：哪一件事能让这个项目 10 倍增长？
5. 标记：哪些东西你在做但其实不重要？
```

**输出重点：**
- 项目的核心闪光点（1-3 个）
- 最关键的瓶颈或突破口
- 应该砍掉或暂停的事项

### 模式 3：User Journey Audit（用户旅程审计）

以新用户身份走一遍完整流程：

```
1. 像新用户一样克隆/安装/体验产品
2. 记录每一个摩擦点（困惑步骤、报错、缺失文档）
3. 给每个步骤计时
4. 与竞品 onboarding 对比
5. 评估 time-to-value（用户多久拿到第一个"赢"？）
6. 推荐：onboarding 前 3 优先级修复项
```

### 模式 4：Feature Prioritization（功能优先级排序）

10 个想法里挑最值得做的 2 个：

```
1. 列出所有候选功能
2. 每个按 ICE 打分：impact(1-5) × confidence(1-5) ÷ effort(1-5)
3. 按 ICE 排序
4. 加入约束： runway、团队规模、依赖关系
5. 输出：带理由的优先级路线图
```

## 输出原则

所有模式都输出**可执行的结论**，不是 essay。
- 每条建议必须有具体的下一步动作
-  Founder Review 必须指出 1-3 个闪光点 + 1 个 10x 机会
- 诊断必须给出 go/no-go 的明确立场，不能模棱两可

## 协作建议

可以与其他 Kimi 技能或工具配合使用：
- 用户旅程审计后，可以用浏览器工具验证实际体验
- 上线后，建议设置监控观察核心指标变化
- 诊断完成后，如需继续产出 PRD/技术方案，请明确告知
